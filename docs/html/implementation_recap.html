1Project Kickoff & Requirements

What: Defined the goal — a Snowflake-best-practice foundation for a fictional electronics retailer (Dark Star Electronics).

Why: A clean foundation makes everything downstream (loaders, dbt models, BI, AI) cheaper and safer. Doing it as code in Git means we can review changes, roll back, and stamp releases.

Non-negotiables

  • Deployable to DEV, TEST, PROD from one codebase
  • Persona roles per environment
  • A starter retail star schema (dimensions + facts)
  • Path to GitHub-driven CI/CD (deferred to a follow-up)

2DCM Project Registration

What: Registered three DCM project objects in Snowflake — one per environment.

Why: Each environment needs its own DCM project so deployments can't accidentally overwrite each other. The DCM project is the Snowflake-side anchor that tracks deployed state and history.
snow dcm create CODE_DB.DCM.DARK_STAR_PROJECT_DEV  -c kb_demo
snow dcm create CODE_DB.DCM.DARK_STAR_PROJECT_TEST -c kb_demo
snow dcm create CODE_DB.DCM.DARK_STAR_PROJECT_PROD -c kb_demo

Initial location was SANDBOX.KB; relocated to CODE_DB.DCM (a curated home for DCM projects) once the layout was clearer.

4Multi-Environment Manifest with Jinja

What: Authored manifest.yml with three targets and a templating block so one set of DEFINE statements becomes three real environments.

Why: Eliminates copy-paste drift between envs and lets us tune per-env knobs (warehouse size, retention) in one place.

Template variables in play

  • {{ db_name }} — environment database

5Infrastructure Definitions

What: One database, four schemas, one warehouse — defined declaratively via DEFINE (not CREATE).

Why: DEFINE is idempotent — DCM diffs the desired state against Snowflake and only emits the deltas. Re-running a deploy is a no-op.
SchemaPurpose
RAWLanding zone for source data
STAGINGCleansed / conformed data
ANALYTICSStar schema (dims + facts)
SERVEBI-friendly views

6Persona-Based Access Control

What: Five database roles per environment, plus one account role for warehouse access. Wired into SYSADMIN.

Why: Database roles are scoped to the database lifecycle — when DEV is dropped, its roles go with it. A separate account role holds warehouse USAGE because, by Snowflake design, database roles cannot.
RoleTypeCapabilities
ADMINDB roleFull DDL + DML
ENGINEERDB roleRead all + write to RAW/STAGING/ANALYTICS
ANALYSTDB roleRead-only on SERVE/ANALYTICS
LOADERDB roleService: write to RAW only
DATA_SCIENTISTDB roleRead all + CREATE in ANALYTICS
DARK_STAR_<ENV>_WAREHOUSE_USERAccount roleUSAGE on env warehouse

A reusable Jinja macro grant_persona_access(db, schema) fans out per-schema USAGE/SELECT grants so the four schemas don't repeat themselves in code.

7Retail Star Schema

What: A starter dim/fact model in ANALYTICS, plus consumption views in SERVE.

Why: Gives the team an immediately-queryable shape that matches how electronics retailers actually report — sales, returns, and inventory anchored to common dimensions.

Dimensions

  • DIM_DATE — full date dimension
  • DIM_CUSTOMER — SCD2-ready customer master
  • DIM_PRODUCT — SKU / category / brand / pricing
  • DIM_STORE — physical, online, and outlet channels
  • DIM_EMPLOYEE — sales associates, linked to a store

Facts

  • FACT_SALES — order-line grain, change-tracked
  • FACT_INVENTORY — daily snapshot per product/store
  • FACT_RETURNS — links back to FACT_SALES

Serve-layer views

  • VW_DAILY_SALES — store × product × day rollup
  • VW_PRODUCT_PERFORMANCE — lifetime sell / return / refund
  • VW_INVENTORY_SNAPSHOT — current stock position

8Validate & Plan

What: Static analysis, then a dry-run plan, before touching live infrastructure.

Why: Catch syntax errors, missing dependencies, and unexpected drops before they hit Snowflake.
snow dcm raw-analyze CODE_DB.DCM.DARK_STAR_PROJECT_DEV -c kb_demo --target DEV
snow dcm plan        CODE_DB.DCM.DARK_STAR_PROJECT_DEV -c kb_demo --target DEV --save-output

Result: analyze passed clean; plan reported 23 CREATEs + 1 ALTER (SYSADMIN) + 0 DROPs per env.

9Deployment to All Three Environments

What: Tagged the deployment v0.1.0 and rolled out to DEV → TEST → PROD.

Why: Aliases give us a clean audit trail and a roll-back target. Promoting in order (DEV → TEST → PROD) is best-practice and lets us spot env-specific issues early.
snow dcm deploy CODE_DB.DCM.DARK_STAR_PROJECT_DEV  --target DEV  --alias v0.1.0
snow dcm deploy CODE_DB.DCM.DARK_STAR_PROJECT_TEST --target TEST --alias v0.1.0

11CI/CD Pipeline (GitHub Actions)

What: Five GitHub Actions workflows wired to per-environment Snowflake service users with RSA key-pair JWT auth, gated by GitHub Environments. PROD requires an explicit reviewer approval before deploying.

Why: A push to main should flow safely through DEV → TEST → PROD without any human running snow dcm deploy, and a leaked DEV key must never be able to touch PROD. Drift detection catches anything that bypasses the pipeline.

Workflows

WorkflowTriggerEnvironmentApproval
dcm_plan.ymlPR to main—Read-only · PR comment
dcm_deploy_dev.ymlpush to maindevnone
dcm_deploy_test.ymlafter DEV successtestnone
dcm_deploy_prod.ymlafter TEST successprodrequired reviewer
dcm_drift.ymlnightly 09:00 UTCper-envopens issue on drift

Auth model

  • Three TYPE = SERVICE Snowflake users — DCM_DEPLOYER_DEV/TEST/PROD — interactive password login disabled.
  • RSA key-pair JWT auth (SNOWFLAKE_JWT); private keys stored only as GitHub Environment secrets.
  • Per-env role owns its database, warehouse, schemas, tables, views, db roles, and the matching WAREHOUSE_USER account role.
  • OIDC federated auth deferred — clean follow-up.
